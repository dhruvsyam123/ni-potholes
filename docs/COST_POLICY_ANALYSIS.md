# Cost Policy Analysis

This note makes the cost argument explicit. The model predicts recurrence risk;
the cost policy decides when it is worth paying extra for a faster or higher
quality repair.

## Decision Rule

For any model threshold:

```text
net value =
  true_positives * intervention_effectiveness * avoided_repeat_cost
  - flagged_defects * incremental_fast_repair_cost
```

Per flagged defect, intervention is justified when:

```text
precision * intervention_effectiveness * avoided_repeat_cost
  > incremental_fast_repair_cost
```

Equivalent:

```text
avoided_repeat_cost / incremental_fast_repair_cost
  > 1 / (precision * intervention_effectiveness)
```

This is where the earlier `1.34x` came from. At threshold `0.60`, precision was
about `0.749`, so if the intervention is 100% effective:

```text
1 / 0.749 = 1.34
```

If the intervention is only 50% effective:

```text
1 / (0.749 * 0.50) = 2.67
```

That statement is mathematically correct, but incomplete: it only tells us the
break-even multiple at one threshold. The better question is which threshold
maximises expected value under realistic cost assumptions.

## Interpretation

The `0.60` threshold is a good general triage threshold, but it is not always
the best economic threshold.

Recommended policy:

- Use `0.55-0.60` if the intervention is a small incremental upgrade over the
  default repair, around `£30-£75` extra per defect.
- Use `0.675-0.70` if the intervention costs around `£75-£125` extra and the
  avoided repeat cost includes inspection/admin plus some deterioration or claim
  risk.
- Use `0.80+` only when the intervention is expensive, around `£300` extra, and
  you want very high precision with low workload.
- Do not justify a full `£1,328` medium patch purely from avoiding one 14-day
  nearby recurrence. That level of work needs a broader asset-management case:
  multi-month recurrence, road safety, traffic management bundling, or preventing
  larger resurfacing/backlog costs.

## Model Implication

For deployment, do not hard-code one threshold globally. Use a policy table:

```text
if incremental_premium <= £50:       act around risk >= 0.55
if incremental_premium £75-£125:     act around risk >= 0.675-0.70
if incremental_premium ~£300:        act around risk >= 0.825
if full patch/resurfacing scale:     require extra evidence beyond this model
```

The next modelling improvement should therefore be a two-stage decision model:

1. Predict recurrence risk.
2. Estimate the appropriate intervention class and incremental cost.

The current model solves stage 1. Stage 2 needs repair-type/cost data that is
not present in the open datasets.

## Dynamic Simulation

The static threshold table above does not let repairs change future events. For
that, see `docs/REPAIR_POLICY_SIMULATION.md` and
`src/simulate_repair_policy.py`.

That simulation processes the future holdout chronologically. When a high-risk
defect is repaired, it probabilistically suppresses future nearby defects. Under
short-term authority-cost assumptions, the simulated saving is about `14-15%`.
Under stronger 30-90 day asset-effect assumptions, the simulated saving is about
`18-23%`.
